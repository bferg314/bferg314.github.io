    SSH Keys: The Magic Lock and Key System

    The World's Best Analogy

    Imagine you have a magic padlock that only you can open — but here's the twist: you can make unlimited copies of that padlock and give them to anyone. Anyone can lock something with one of your padlocks, but only you can unlock it, because only you have the one true key.

    That's exactly how SSH keys work.

    • Your private key = the one true key you keep in your pocket forever
    • Your public key = the padlock you hand out freely to every server you want to log into

    When you try to connect to a server, it "locks a challenge" with your padlock and says: "If you're really you, prove it — unlock this." Your private key does it instantly. No password needed. No way to fake it.

    The .ssh Directory: Your Key Ring

    On your computer, everything SSH-related lives in one folder:

    ~/.ssh/

    (~ means your home directory — /home/yourname on Linux/Mac, C:\Users\yourname on Windows)

    Here's what you'll find inside:

    ~/.ssh/
├── id_ed25519          ← Your PRIVATE key (guard this with your life)
├── id_ed25519.pub      ← Your PUBLIC key (hand this out freely)
├── authorized_keys     ← Public keys of people allowed into THIS machine
├── known_hosts         ← Fingerprints of servers you've connected to before
└── config              ← Shortcuts and settings for SSH connections

    The Files, Explained

    File What it is Who sees it
    id_ed25519 Your private key Only you. Ever.
    id_ed25519.pub Your public key Anyone — servers, teammates, GitHub
    authorized_keys List of public keys allowed in to this machine Lives on the server
    known_hosts "I've seen this server before and trust it" Your machine only
    config Saved shortcuts (aliases, ports, usernames) Your machine only

    Step 1: Creating Your Keys

    If you don't have keys yet, generate them with one command:

    ssh-keygen -t ed25519 -C "your-email@example.com"
    • -t ed25519 — use the modern, fast, secure key type (preferred over the older rsa)
    • -C "..." — a comment so you know which key is which (your email works great)

    It will ask: 1. Where to save it — just hit Enter to accept the default (~/.ssh/id_ed25519) 2. A passphrase — this is a password that encrypts your private key on disk. Use one. It's your last line of defense if someone steals your laptop.

    Now you have two files: - ~/.ssh/id_ed25519 — private, never share - ~/.ssh/id_ed25519.pub — public, share everywhere

    To see your public key: 

    cat ~/.ssh/id_ed25519.pub

    It looks like a long string: ssh-ed25519 AAAAC3Nza... your-email@example.com

    Step 2: Getting Your Public Key onto a Server

    To log into a server without a password, the server needs your public key in its ~/.ssh/authorized_keys file.

    The Easy Way (if you can already log in with a password)

    ssh-copy-id user@server-address

    This command does everything for you — connects with your password, then drops your public key into the right place on the server.

    The Manual Way (if ssh-copy-id isn't available)

    1. Copy your public key to your clipboard: 

      cat ~/.ssh/id_ed25519.pub
# Copy the output

    2. Log into the server (with password this time): 

      ssh user@server-address

    3. On the server, paste your key into authorized_keys: 

      mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "PASTE_YOUR_PUBLIC_KEY_HERE" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

    Or if the file already exists, just open it in an editor and paste your public key in as a new line — one key per line: 

    nano ~/.ssh/authorized_keys
# or: vim ~/.ssh/authorized_keys

    1. Log out and reconnect — no password this time: 
      ssh user@server-address

    For GitHub / GitLab / Bitbucket

    1. Copy your public key: cat ~/.ssh/id_ed25519.pub
    2. Go to your account settings → SSH Keys
    3. Paste it in
    4. Done — git push will now work without a password

    Step 3: The config File — Your Shortcut Book

    The ~/.ssh/config file lets you define aliases for servers so you don't have to remember IPs, ports, and usernames.

    # ~/.ssh/config

Host myserver
    HostName 192.168.1.100
    User deploy
    IdentityFile ~/.ssh/id_ed25519

Host github
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519

Host jumpbox
    HostName bastion.example.com
    User ec2-user
    Port 2222
    IdentityFile ~/.ssh/id_ed25519

    Now instead of: 

    ssh deploy@192.168.1.100 -i ~/.ssh/id_ed25519

    You just type: 

    ssh myserver

    Guarding Your Private Key: The Golden Rules

    Your private key is the whole game. If someone gets it, they can be you everywhere your public key lives.

    Rule 1: Never share it. Not to a coworker. Not to a sysadmin. Not by email. Not by Slack. Never copy-paste it anywhere. The only place id_ed25519 should ever exist is on your machine(s) that you personally control.

    Rule 2: Use a passphrase. When you generate your key, set a passphrase. This encrypts the key file itself. Even if someone steals your laptop, they'd need to crack the passphrase to use the key.

    Rule 3: Use ssh-agent so you only type the passphrase once per session.

    # Start the agent (usually automatic)
eval "$(ssh-agent -s)"

# Add your key (type passphrase once)
ssh-add ~/.ssh/id_ed25519

# Now SSH everywhere without typing the passphrase again until you reboot

    Rule 4: Check your permissions. SSH will refuse to work if your key files are readable by others. Fix permissions with:

    chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
chmod 600 ~/.ssh/authorized_keys
chmod 600 ~/.ssh/config

    Rule 5: One key per device, not one key everywhere. Generate a separate key on each machine you own. That way if one machine is compromised, you revoke just that key — not all access to everything everywhere.

    The known_hosts File: Trusting Servers

    The first time you connect to a server, SSH shows you its "fingerprint" — a unique ID:

    The authenticity of host 'example.com (93.184.216.34)' can't be established.
ED25519 key fingerprint is SHA256:abc123xyz...
Are you sure you want to continue connecting? (yes/no)?

    Type yes, and SSH saves that fingerprint in ~/.ssh/known_hosts. Next time you connect, SSH checks: "Is this the same server I trusted before?" If the fingerprint changed (could be a server rebuild, or could be an attacker intercepting your connection), SSH screams a warning and refuses to connect.

    Don't blindly ignore that warning. It exists for a reason.

    To remove a stale entry (e.g., after rebuilding a server): 

    ssh-keygen -R server-address

    Quick Reference Cheat Sheet

    # Generate a new key pair
ssh-keygen -t ed25519 -C "me@example.com"

# See your public key (safe to share)
cat ~/.ssh/id_ed25519.pub

# Copy your public key to a server
ssh-copy-id user@server

# Connect to a server
ssh user@server

# Connect using a specific key
ssh -i ~/.ssh/id_ed25519 user@server

# Add key to agent (so you don't retype passphrase)
ssh-add ~/.ssh/id_ed25519

# Remove a stale known_hosts entry
ssh-keygen -R server-address

# Fix permissions (do this if SSH complains)
chmod 700 ~/.ssh && chmod 600 ~/.ssh/id_ed25519

    The 10-Second Summary

    1. Private key: your secret. Lives only on your machine. Never share it.
    2. Public key: your padlock. Copy it to every server or service you want access to.
    3. authorized_keys: the server's guest list. Your public key goes here.
    4. config: your shortcut book. Makes ssh myserver work instead of typing everything out.
    5. known_hosts: your memory of servers you've trusted before. SSH checks this automatically.

    The whole system is: hand out your padlock, keep your key, prove who you are without ever revealing the key itself. Elegant, fast, and nearly impossible to fake.
