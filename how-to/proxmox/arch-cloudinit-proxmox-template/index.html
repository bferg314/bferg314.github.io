    Arch Linux template for Proxmox with cloud-init

    Build a reusable VM template from a fresh Arch cloud image. Every clone inherits SSH keys, hostname, and user config automatically. The whole process takes about 10 minutes.

    Prerequisites

    • Proxmox VE 7.x or 8.x or 9.x
    • Shell access to the Proxmox host
    • A storage pool with enough space (20+ GB recommended)
    • An SSH public key ready to inject

    Phase 1 — Download the Arch cloud image

    Run these commands on the Proxmox host shell.

    # Create a working directory
mkdir -p /var/lib/vz/template/iso
cd /var/lib/vz/template/iso

# Download the latest Arch cloud image
wget https://geo.mirror.pkgbuild.com/images/latest/Arch-Linux-x86_64-cloudimg.qcow2

    Mirror

    geo.mirror.pkgbuild.com resolves to a regionally close mirror automatically. You can also pull from archive.archlinux.org/iso/latest/ if you want a pinned version.

    Verify the checksum

    wget https://geo.mirror.pkgbuild.com/images/latest/Arch-Linux-x86_64-cloudimg.qcow2.SHA256
sha256sum -c Arch-Linux-x86_64-cloudimg.qcow2.SHA256

    Phase 2 — Create the base VM

    Pick a VM ID that won't conflict with existing machines. 9000+ is a common convention for templates.

    qm create 9000 \
  --name arch-template \
  --memory 2048 \
  --cores 2 \
  --net0 virtio,bridge=vmbr0 \
  --ostype l26 \
  --agent 1

    --agent 1

    This enables QEMU guest agent support on the VM record. You still need to install and enable qemu-guest-agent inside the VM before converting to template — covered in Phase 4.

    Import the disk

    Import the qcow2 image into your Proxmox storage pool. Replace local-lvm with your actual storage name if different.

    qm importdisk 9000 \
  /var/lib/vz/template/iso/Arch-Linux-x86_64-cloudimg.qcow2 \
  local-lvm

    Phase 3 — Attach and configure the disk

    Attach as VirtIO SCSI

    The imported disk shows up as unused0. Attach it with VirtIO SCSI for best performance, then add a cloud-init drive on the same controller.

    # Attach the main disk
qm set 9000 --scsihw virtio-scsi-pci --scsi0 local-lvm:vm-9000-disk-0

# Add a cloud-init drive (IDE2 is conventional)
qm set 9000 --ide2 local-lvm:cloudinit

# Set boot order to the SCSI disk
qm set 9000 --boot c --bootdisk scsi0

# Enable serial console (cloud-init needs this to output status)
qm set 9000 --serial0 socket --vga serial0

    --vga serial0

    Arch cloud images output their boot console to the serial port by default. Without this flag the Proxmox console tab shows a blank screen during first boot.

    Resize the disk

    The cloud image ships as ~2 GB. Resize it now so all clones inherit the larger size.

    # Resize to 20 GB — adjust to taste
qm resize 9000 scsi0 20G

    Resize before templating

    You cannot resize a disk after a VM is converted to a template. Do it here, not after Phase 5.

    Phase 4 — Configure cloud-init

    Set cloud-init defaults

    These values become the defaults for every clone. You can override them per-clone later from the UI or CLI.

    qm set 9000 \
  --ciuser your-username \
  --sshkeys ~/.ssh/id_rsa.pub \
  --ipconfig0 ip=dhcp \
  --nameserver 1.1.1.1

    Avoid setting --cipassword

    Setting a password in the cloud-init config stores it in plain text on the Proxmox host and bakes it into every clone. Prefer SSH key authentication only. If you need a password after first boot, SSH in and set one manually:

    sudo passwd your-username

    If you do need a password in the template (e.g., for console access without SSH), you can add --cipassword 'your-password' to the command above — just be aware of the trade-off.

    Cloud-init variable reference

    Variable Description
    --ciuser Default user created on first boot
    --cipassword Password for the cloud-init user
    --sshkeys Public key(s) added to authorized_keys
    --ipconfig0 ip=dhcp or ip=192.168.1.x/24,gw=...
    --nameserver DNS resolver for the VM
    --searchdomain DNS search domain (optional)

    Boot and prep the VM

    Start the VM once to install the guest agent and apply any base packages before freezing it as a template.

    # Start the VM
qm start 9000

# SSH in once it boots (check IP in the Proxmox summary tab)
ssh your-username@<vm-ip>

    Once inside the VM:

    # Update and install the guest agent
sudo pacman -Syu --noconfirm
sudo pacman -S --noconfirm qemu-guest-agent

# Enable and start it
sudo systemctl enable --now qemu-guest-agent

# Clean up package cache to keep the template lean
sudo pacman -Scc --noconfirm

# Clear machine-id so clones each generate a unique one
sudo truncate -s 0 /etc/machine-id
sudo truncate -s 0 /var/lib/dbus/machine-id

# Remove SSH host keys (cloud-init regenerates them per clone)
sudo rm -f /etc/ssh/ssh_host_*

# Clear cloud-init state so it reruns on clone first boot
sudo cloud-init clean --logs

# Shut down
sudo poweroff

    machine-id is important

    If you skip clearing /etc/machine-id, every clone will present the same ID to your DHCP server and they will fight over the same IP address. The truncate approach (rather than deleting the file) works because systemd regenerates the ID on boot only when the file exists but is empty.

    Phase 5 — Convert to template

    This is a one-way operation. The VM becomes read-only and can only be cloned, not started directly.

    qm template 9000

    Phase 6 — Clone and verify

    Clone the template

    A full clone creates a completely independent VM. A linked clone shares the template disk (faster, but depends on the template staying intact).

    # Full clone — recommended for production VMs
qm clone 9000 101 --name my-arch-vm --full

# Override cloud-init values per-clone
qm set 101 --ciuser admin --ipconfig0 ip=dhcp

# Start it
qm start 101

    Full vs linked clone

    Linked clones provision in seconds because they only store the delta against the template disk. For ephemeral dev VMs, linked is fine. For anything long-term, use full so you can modify or delete the template later without affecting running VMs.

    Verify cloud-init ran correctly

    SSH in after first boot and check that cloud-init completed without errors.

    # Check cloud-init final status
cloud-init status --long

# Confirm a unique machine-id was generated
cat /etc/machine-id

# Confirm SSH host keys were regenerated
ls /etc/ssh/ssh_host_*

# Confirm guest agent is running
systemctl status qemu-guest-agent

    Using the Proxmox web UI

    You can do the clone and cloud-init overrides entirely from the browser: right-click the template in the sidebar, choose Clone, then open the Cloud-Init tab on the new VM before starting it.

    Pre-flight checklist

    • [ ] Downloaded and verified the Arch cloud image
    • [ ] Created VM with VirtIO SCSI and cloud-init drive on ide2
    • [ ] Resized disk before templating
    • [ ] Installed and enabled qemu-guest-agent
    • [ ] Cleared machine-id, SSH host keys, and cloud-init state
    • [ ] Converted to template with qm template
    • [ ] Cloned and verified cloud-init status on first boot
